# Flux GitOps Migration Design

## Goal

Replace the current manual `helm upgrade` deployment workflow and Terraform-managed in-cluster resources with Flux CD GitOps. Terraform retains ownership of cloud resources only. All in-cluster state is managed declaratively via Git.

## Directory Structure

```
clusters/production/
├── flux-system/                    # Auto-generated by flux bootstrap
│   ├── gotk-components.yaml
│   ├── gotk-sync.yaml
│   └── kustomization.yaml
├── kustomization.yaml              # Root Kustomize entrypoint for Flux bootstrap
├── infrastructure.yaml             # Flux Kustomization (no dependencies)
├── cert-manager-issuers.yaml       # Flux Kustomization (dependsOn: infrastructure)
├── monitoring.yaml                 # Flux Kustomization (optional, suspended by default)
├── apps.yaml                       # Flux Kustomization (dependsOn: cert-manager-issuers)
├── infrastructure/
│   ├── kustomization.yaml          # Kustomize entrypoint for infrastructure resources
│   ├── sources/
│   │   ├── helmrepo-ingress-nginx.yaml
│   │   ├── helmrepo-jetstack.yaml
│   │   └── helmrepo-external-dns.yaml
│   ├── ingress-nginx.yaml          # HelmRelease
│   ├── cert-manager.yaml           # HelmRelease
│   ├── external-dns.yaml           # HelmRelease
│   ├── namespaces.yaml             # ingress-nginx + sub2api namespaces
│   └── issuers/
│       ├── kustomization.yaml
│       └── cluster-issuer.yaml     # Raw ClusterIssuer manifest
├── monitoring/
│   ├── kustomization.yaml          # Kustomize entrypoint for monitoring resources
│   ├── namespace.yaml              # monitoring namespace
│   └── monitoring.yaml             # HelmRelease via GitRepository
└── apps/
    ├── kustomization.yaml          # Kustomize entrypoint for app resources
    └── sub2api.yaml                # HelmRelease via GitRepository
```

Dependency chain: infrastructure -> cert-manager-issuers -> apps. Monitoring is an optional sibling Kustomization that can be unsuspended after its values and secrets are ready.

## Sources

### Upstream HelmRepositories (third-party charts)

| Name | URL | Used by |
|------|-----|---------|
| ingress-nginx | `https://kubernetes.github.io/ingress-nginx` | ingress-nginx HelmRelease |
| jetstack | `https://charts.jetstack.io` | cert-manager HelmRelease |
| external-dns | `https://kubernetes-sigs.github.io/external-dns` | external-dns HelmRelease |

### GitRepository (in-repo charts)

The `flux-system` GitRepository created by `flux bootstrap` points at this repo. The monitoring and sub2api HelmReleases reference it with chart paths `deploy/helm/monitoring` and `deploy/helm/sub2api` respectively.

Both Git-backed HelmReleases should use `reconcileStrategy: Revision` so changes under `deploy/helm/...` produce a new chart artifact even when `Chart.yaml` version is unchanged.

## HelmRelease Configurations

### ingress-nginx

- **Source**: ingress-nginx HelmRepository
- **Chart version**: 4.12.1
- **Namespace**: ingress-nginx
- **Values**:
  - `controller.service.externalTrafficPolicy: Local`
  - `controller.service.annotations."service.beta.kubernetes.io/do-loadbalancer-name": sub2api-lb`
  - `controller.service.annotations."service.beta.kubernetes.io/do-loadbalancer-tls-passthrough": "true"`
  - `controller.config.enable-underscores-in-headers: "true"`

### cert-manager

- **Source**: jetstack HelmRepository
- **Chart version**: 1.17.1
- **Namespace**: cert-manager
- **Values**:
  - `crds.enabled: true`

### ClusterIssuer (infrastructure/issuers/cluster-issuer.yaml)

Raw Kubernetes manifest (not a HelmRelease) applied by a dedicated Flux `Kustomization` that depends on the infrastructure layer. This makes the first reconciliation deterministic by waiting for cert-manager and its CRDs before applying the issuer.

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: <letsencrypt_email>
    privateKeySecretRef:
      name: letsencrypt-prod
    solvers:
      - dns01:
          cloudflare:
            apiTokenSecretRef:
              name: cloudflare-api-token
              key: api-token
```

### external-dns

- **Source**: external-dns HelmRepository
- **Chart version**: 1.16.1
- **Namespace**: external-dns
- **Values**:
  - `provider.name: cloudflare`
  - `policy: sync`
  - `txtOwnerId: <cluster_name>`
  - `domainFilters[0]: <domain_suffix>`
  - `sources[0]: ingress`
  - Environment variable `CF_API_TOKEN` from `cloudflare-api-token` Secret

### monitoring

- **Source**: flux-system GitRepository, path `deploy/helm/monitoring`
- **Namespace**: monitoring
- **valuesFrom**: `monitoring-secrets` Secret (Grafana admin password, R2 credentials, Grafana DB credentials)
- **Inline values**: Grafana ingress host, R2 endpoints, bucket names (non-sensitive)

### sub2api

- **Source**: flux-system GitRepository, path `deploy/helm/sub2api`
- **Namespace**: sub2api
- **valuesFrom**: `sub2api-secrets` Secret (JWT, TOTP, admin password, DB/Redis passwords)
- **Inline values**: image tags, replica counts, resource limits, ingress hosts, autoscaling config

## Reconciliation Settings

- All HelmReleases: `interval: 10m`
- GitRepository: `interval: 1m`
- Install remediation: `retries: 3`
- Upgrade remediation: `retries: 3`, `remediateLastFailure: true` (automatic rollback on upgrade failure)

## Secrets Management

Secrets are created manually in-cluster. HelmReleases reference them via `valuesFrom`.

### Pre-created Secrets

| Secret | Namespace | Contents | Created by |
|--------|-----------|----------|------------|
| `cloudflare-api-token` | cert-manager | Cloudflare API token | Terraform (kept) |
| `cloudflare-api-token` | external-dns | Cloudflare API token | Terraform (kept) |
| `monitoring-secrets` | monitoring | Grafana admin password, R2 access key, R2 secret key, Grafana DB user/password | Manual (kubectl) |
| `sub2api-secrets` | sub2api | JWT secret, TOTP key, admin password, bundled DB/Redis passwords, external DB/Redis passwords | Manual (kubectl) |

Future improvement: encrypt secrets with SOPS and commit to Git for full GitOps. The `valuesFrom` pattern remains the same.

## Terraform Scope After Migration

### Keeps

- `module.doks` — DOKS cluster provisioning
- `module.database` — DO Managed PostgreSQL (optional)
- `module.storage` — Cloudflare R2 buckets (optional)
- Cloudflare API token Secrets in `cert-manager` and `external-dns` namespaces

### Removes

- `module.kubernetes` — ingress-nginx, cert-manager, ExternalDNS, ClusterIssuer, namespaces (moved to Flux infrastructure layer)
- `module.monitoring` — entire LGTM stack (moved to Flux monitoring layer)
- Helm provider (no longer needed for Terraform, Kubernetes provider still needed for secrets)
- Monitoring-related variables: `enable_monitoring`, `grafana_hostname_prefix`, `grafana_admin_password`, R2 credential passthrough, Grafana DB credential passthrough
- Monitoring-related locals, checks, and `random_password.grafana_admin_password`

### Updated outputs

Remove: load balancer IP output (was read from ingress-nginx service managed by Terraform).

## Migration Procedure

1. Run `flux bootstrap github` to install Flux controllers and create the `flux-system` source
2. Create manual Secrets (`monitoring-secrets`, `sub2api-secrets`) in cluster
3. Commit all Flux manifests to `clusters/production/`
4. Flux adopts existing Helm releases by matching release name and namespace — no reinstall, no downtime
5. Verify all HelmReleases show `Ready: True` via `flux get helmreleases -A`
6. Run `terraform state rm` for each adopted resource (ingress-nginx, cert-manager, external-dns, monitoring Helm releases, namespaces, ClusterIssuer)
7. Remove `module.kubernetes` and `module.monitoring` from `infra/production/main.tf`
8. Remove unused variables, locals, checks, and outputs
9. Run `terraform plan` to confirm no changes
10. Update DEPLOY.md to reflect the new Flux-based workflow

## Deployment Workflow

### Deploying a new application version

```
git tag v0.3.0 → GitHub Actions builds images → developer edits image tags 
in clusters/production/apps/sub2api.yaml → git commit & push → Flux syncs
```

### Rolling back

```
git revert the image tag commit → push → Flux rolls back
```

### Upgrading infrastructure components

Edit the chart `version` in the relevant HelmRelease YAML, commit, push.

### Checking status

```bash
flux get helmreleases -A
flux get kustomizations -A
kubectl get helmreleases -A
```
